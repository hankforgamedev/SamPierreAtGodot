# Sam Pierre 遊戲可行性評估

根據〈Sam Pierre 第一次試玩遊戲心得〉與目前 Godot 專案狀態整理。

## 結論

大部分想法都可行，真正的瓶頸不是功能本身，而是資料結構還不夠穩。

目前已經能做到的基礎：

- 章節切換
- 對話顯示
- 選項分支
- 分支後回主線

因此，像「ABC 三個選項，各自顯示後果，再回主線」這件事，本身不是問題。現在真正缺的是：

1. 可持續保存的選擇後果
2. 劇本文字與程式邏輯的拆分
3. 更方便的回看與編修流程

## 可直接做

### 1. Scroll-up / go back function

可行，而且優先度高。

這是純介面層功能，不需要先重寫整個劇情系統。可以先做：

- 對話歷史紀錄
- 回看上一句
- 顯示近期數句內容

### 2. 每個章節獨立

可行，且目前架構本來就接近這個方向。

建議維持：

- 一章一份資料
- 章內自行處理分支
- 章與章之間只用少量旗標互相影響

### 3. 攻擊的決策太倉促

這是內容節奏問題，不是技術問題。

適合立刻修，方式是補：

- 動手前的心理鋪陳
- 選項前的壓力累積
- 選完後的短反應或餘波

### 4. 小改小修要容易編輯

可行，但前提是先把文字抽出程式碼。

這件事做完之後，Obsidian 編修和 AI 協作才會真的順。

## 可做，但要先補架構

### 5. text story 和 code decoupled

現在是半解耦，不是完全解耦。

現況大致是：

- 畫面邏輯在 Godot script
- 劇本文字也還寫在 Godot script 裡

這代表：

- 改字不方便
- AI 容易改到不該改的程式
- Obsidian workflow 很難直接接上

建議下一步：

- 把章節文字抽到外部資料檔
- Godot 只負責讀資料與顯示

格式可選：

- JSON：最好解析，最穩
- YAML：比較好讀，但 parser 要注意
- Markdown + 自訂標記：最貼近 Obsidian，但規格要先收斂

如果目標是「我在 Obsidian 編輯，AI 可以直接匯入」，那最實際的做法是：

Obsidian 寫 Markdown -> 轉換成 Godot 可讀章節資料 -> 遊戲載入

### 6. 關鍵影響

像：

- 有無跟老媽吵架
- 有無推人

這些都可行，但要先加全域旗標系統。

例如：

- `argued_with_mom = true`
- `pushed_someone = true`

之後章節再根據旗標決定：

- 額外台詞
- 額外場景
- 選項是否出現
- 小遊戲是否開啟

### 7. interchapter / intrachapter

可做，但要先定義清楚。

建議不要一開始就做大規模分歧劇情樹，否則編修成本會很快失控。

比較穩的方式是：

- 章內分支負責當下情緒與局部後果
- 章間旗標負責長期回應
- 主線仍維持收束

## 先不要急著做

### 8. Assets problem

目前不是第一優先。

先把：

- 劇本流程
- 資料格式
- 分支與旗標
- 編修流程

穩定下來，再補資產比較合理。

### 9. 不要做成視覺小說

這是方向問題，不是單一功能問題。

如果只是維持目前的對話框、選項、章節推進，那本質上還是很接近視覺小說。

如果真的不要落到純 VN，後面就要補至少一種非純閱讀玩法，例如：

- 公務員小遊戲
- 簡單探索
- 場景互動
- 狀態管理

### 10. 選擇不跳脫輪迴：玩公務員小遊戲

可行，但這已經不是小修，是新增玩法模組。

建議排在以下事項之後：

1. 外部文本格式
2. 旗標系統
3. 回看功能

等這些穩了，再做小遊戲才不會反覆返工。

## 平台判斷：HTML or Godot

現階段建議繼續用 Godot，不建議切去 HTML。

理由：

- 目前已經有 Godot 專案基礎
- 章節、選項、場景切換已經存在
- 後續如果要加小遊戲、互動、演出，Godot 比較自然

如果切去 HTML，短期可能讓文字流程變順，但整體遊戲結構會更鬆，之後補玩法反而更麻煩。

## 實作優先順序

### 第一優先

把劇本文字從 GDScript 抽出去。

### 第二優先

加入 flags / consequences system。

### 第三優先

補 scroll-back / dialogue history。

### 第四優先

再決定是否加入公務員小遊戲，讓作品開始脫離純 VN 形式。

## 建議的下一步

最實際的開發順序：

1. 設計一個 Godot 可讀的章節資料格式
2. 讓 Obsidian 筆記可轉成該格式
3. 在遊戲裡加入 flags
4. 在 UI 補回看功能
5. 再開始做章間影響與額外玩法

## 補充

- 「他會亂改劇本」這個問題，不應靠 prompt 單獨解決，應靠資料與流程隔離解決。
- 最好的做法不是一直提醒 AI 不要改，而是讓 AI 沒那麼容易碰到不該改的部分。（或根本不要碰到）
- 換句話說，文本、邏輯、資產、轉換腳本要分層。
